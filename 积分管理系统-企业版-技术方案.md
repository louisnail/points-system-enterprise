## 一、项目概述

### 1.1 项目背景
- **项目名称**：积分管理系统-企业版
- **目标用户**：TD、XD、TY、WB 四公司融合人员（约300人）
- **核心目标**：建立跨主体的统一积分评价体系，支持过程分+结果分双轨管理

### 1.2 核心特点
| 特点 | 说明 |
|------|------|
| 多主体融合 | 同盾(TD)、小盾(XD)、同彦(TY)、WB 人员统一排名，标记归属 |
| 过程分机制 | 每月更新，历史存档，当月有效 |
| 多榜单体系 | 按岗位分榜单，支持跨部门公共榜单 |
| 季度之星 | 人工评选，独立于自动排名 |
| 第三方接入 | 预留业务系统推送接口 |

## 二、技术架构

### 2.1 技术栈（企业版实际）

| 层级 | 技术选型 | 版本 |
|------|---------|------|
| **前端** | **Vue 3 + Ant Design Vue + Vite** | Vue 3.5+ |
| 后端 | NestJS + TypeScript | 10.x |

#### 3.1.1 员工表 (user)
```sql
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `employee_no` VARCHAR(20) NOT NULL COMMENT '工号',
  `name` VARCHAR(50) NOT NULL COMMENT '姓名',
  `display_name` VARCHAR(100) NOT NULL COMMENT '显示名称(姓名+工号)',
  `gender` TINYINT COMMENT '性别:1男2女',
  `hire_date` DATE COMMENT '入职日期',
  `email` VARCHAR(100) COMMENT '公司邮箱',
  `education` VARCHAR(20) COMMENT '最高学历',
  `company_belong` VARCHAR(20) NOT NULL COMMENT '归属组织:TD/XD/TY/WB',
  `base_location` VARCHAR(50) COMMENT 'base地',
  `department_id` BIGINT NOT NULL COMMENT '部门ID',
  `position` VARCHAR(100) COMMENT '岗位',
  `rank_level` VARCHAR(10) COMMENT '职级:P3-P12,M1-M6',
  `manager_id` BIGINT COMMENT '主管ID',
  `project_manager_id` BIGINT COMMENT '项目经理ID',
  `project_name` VARCHAR(200) COMMENT '所在项目',
  `project_role` VARCHAR(50) COMMENT '项目角色',
  `ranking_list_id` BIGINT COMMENT '所属榜单ID(nullable,允许未分配榜单)',
  `total_points` DECIMAL(10,1) DEFAULT 0 COMMENT '当前总积分',
  `ranking` INT COMMENT '当前排名',
  `honors` JSON COMMENT '历史荣誉记录',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态:1在职2禁用3离职4待岗5停薪留职',
  `is_ranking_disabled` TINYINT DEFAULT 0 COMMENT '是否禁用排名:0否1是',
  `password_hash` VARCHAR(255) NOT NULL COMMENT '密码哈希',
  `role` VARCHAR(20) NOT NULL DEFAULT 'employee' COMMENT '角色:super_admin/dept_admin/employee',
  `custom_fields` JSON COMMENT '自定义字段数据(动态扩展)',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_employee_no` (`employee_no`),
  KEY `idx_department` (`department_id`),
  KEY `idx_status` (`status`),
  KEY `idx_ranking_list` (`ranking_list_id`)
) ENGINE=InnoDB COMMENT='员工表';
```

> **v3.0 变更**：`ranking_list_id` 改为 nullable（允许未分配榜单）；状态值2从"冻结"改为"禁用"；新增 `password_hash`（认证登录）、`role`（角色标识）、`custom_fields`（动态自定义字段 JSON）三列。

#### 3.1.11 积分画像维度表 (score_profile_dimension)

```sql
CREATE TABLE `score_profile_dimension` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `template_id` BIGINT NOT NULL COMMENT '所属模板ID',
  `name` VARCHAR(100) NOT NULL COMMENT '维度名称',
  `source_type` VARCHAR(20) NOT NULL COMMENT '数据来源:category/indicator/custom',
  `source_id` BIGINT COMMENT '来源ID(分类ID或指标ID)',
  `sort_order` INT DEFAULT 0 COMMENT '排序',
  `ref_type` VARCHAR(20) DEFAULT 'average' COMMENT '参考值类型:fixed/average/max',
  `ref_value` DECIMAL(10,1) COMMENT '固定参考值(ref_type=fixed时)',
  `ref_scope` VARCHAR(20) DEFAULT 'all' COMMENT '参考值计算范围:all/ranking_list',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  KEY `idx_template_sort` (`template_id`, `sort_order`),
  CONSTRAINT `fk_dim_template` FOREIGN KEY (`template_id`) REFERENCES `score_profile_template`(`id`) ON DELETE CASCADE
) ENGINE=InnoDB COMMENT='积分画像维度表';
```

**维度数据来源**：
- `category`：管控项（取该分类下所有指标的积分合计）
- `indicator`：单指标（取该指标的积分值）
- `custom`：自定义维度（手动设定参考值）

**参考值类型**：
- `average`：全员/同榜单平均值
- `max`：全员/同榜单最大值
- `fixed`：管理员手动设定的固定参考值

#### 3.1.12 自定义字段定义表 (custom_field_definition)

> **v3.0 新增** — 支持员工管理和指标管理的动态自定义字段

```sql
CREATE TABLE `custom_field_definition` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `target_module` VARCHAR(30) NOT NULL COMMENT '目标模块:user/indicator',
  `field_key` VARCHAR(50) NOT NULL COMMENT '字段标识(后端自动生成cf_{timestamp})',
  `display_name` VARCHAR(100) NOT NULL COMMENT '字段显示名称',
  `field_type` VARCHAR(20) NOT NULL COMMENT '字段类型:text/number/date/select',
  `options` JSON COMMENT '下拉选项(field_type=select时)',
  `is_required` TINYINT DEFAULT 0 COMMENT '是否必填',
  `is_visible` TINYINT DEFAULT 1 COMMENT '是否可见',
  `sort_order` INT DEFAULT 0 COMMENT '排序序号',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY `idx_module_key` (`target_module`, `field_key`)
) ENGINE=InnoDB COMMENT='自定义字段定义表';
```

**自定义字段与实体的关系**：自定义字段定义存于此表，具体数据值存于目标实体的 `custom_fields` JSON 列中（user.custom_fields / indicator.custom_fields），key 为 `field_key`。

## 五、核心业务逻辑

### 5.1 过程分更新逻辑

```
每月维护流程:
1. 专人上传Excel或手动录入过程分
2. 系统对比: 被评价人 + 指标项 + 归属月份
3. 如存在上月记录 → 更新分值,历史存入process_point_history
4. 如不存在 → 新建记录
5. 当月排名计算时,只取当月最新过程分

计算公式:
人员当前总分 = 所有结果分总和 + 当月过程分最新值
```

### 5.2 排名计算逻辑

```
按需触发（"重新计算排名"按钮）或每日定时任务:
1. 按榜单分组计算
2. 对每个榜单内人员:
   - 计算结果分总和
   - 查询当月过程分
   - 总分 = 结果分 + 过程分
3. Redis缓存排名结果
4. 更新user表ranking字段
```

### 5.3 季度之星发布流程

```
1. 管理员选择季度
2. 点击"选定人员"逐一添加（指定榜单、排序、点评）
3. 可编辑展示顺序和点评内容
4. 点击"发布"将该季度之星公开
5. 发布后页面显示绿色成功提示
6. 可"撤回发布"回到编辑状态
```

## 十、注意事项

1. **数据安全**: 员工积分数据敏感，需做好权限控制（参见 4.6 权限与数据隔离规则）
2. **并发处理**: 排名计算时锁定，避免并发问题
3. **备份策略**: 数据库每日自动备份
4. **日志记录**: 关键操作（积分修改、审核、已审核积分变更）需记录操作日志
5. **性能优化**: 排名数据Redis缓存，数据库加索引
6. **驾驶舱性能**: 统计类查询使用物化视图或定时预计算，避免实时大量聚合查询
7. **AI 接口安全**: Phase 5 接入 LLM 时，需对传入数据脱敏，避免泄露员工隐私
