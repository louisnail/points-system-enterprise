## 一、项目概述

### 1.1 项目背景
- **项目名称**：积分管理系统-企业版
- **目标用户**：TD、XD、TY、WB 四公司融合人员（约300人）
- **核心目标**：建立跨主体的统一积分评价体系，支持过程分+结果分双轨管理

### 1.2 核心特点
| 特点 | 说明 |
|------|------|
| 多主体融合 | 同盾(TD)、小盾(XD)、同彦(TY)、WB 人员统一排名，标记归属 |
| 过程分机制 | 每月更新，历史存档，当月有效 |
| 多榜单体系 | 按岗位分榜单，支持跨部门公共榜单 |
| 季度之星 | 人工评选，独立于自动排名 |
| 第三方接入 | 预留业务系统推送接口 |

## 三、数据库设计

### 3.1 核心表结构

> 与原始技术方案 v1.0 一致，此处保留完整定义。

#### 3.1.1 员工表 (user)
```sql
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `employee_no` VARCHAR(20) NOT NULL COMMENT '工号',
  `name` VARCHAR(50) NOT NULL COMMENT '姓名',
  `display_name` VARCHAR(100) NOT NULL COMMENT '显示名称(姓名+工号)',
  `gender` TINYINT COMMENT '性别:1男2女',
  `hire_date` DATE COMMENT '入职日期',
  `email` VARCHAR(100) COMMENT '公司邮箱',
  `education` VARCHAR(20) COMMENT '最高学历',
  `company_belong` VARCHAR(20) NOT NULL COMMENT '归属组织:TD/XD/TY/WB',
  `base_location` VARCHAR(50) COMMENT 'base地',
  `department_id` BIGINT NOT NULL COMMENT '部门ID',
  `position` VARCHAR(100) COMMENT '岗位',
  `rank_level` VARCHAR(10) COMMENT '职级:P3-P12,M1-M6',
  `manager_id` BIGINT COMMENT '主管ID',
  `project_manager_id` BIGINT COMMENT '项目经理ID',
  `project_name` VARCHAR(200) COMMENT '所在项目',
  `project_role` VARCHAR(50) COMMENT '项目角色',
  `ranking_list_id` BIGINT NOT NULL COMMENT '所属榜单ID',
  `total_points` DECIMAL(10,1) DEFAULT 0 COMMENT '当前总积分',
  `ranking` INT COMMENT '当前排名',
  `honors` JSON COMMENT '历史荣誉记录',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态:1在职2冻结3离职4待岗5停薪留职',
  `is_ranking_disabled` TINYINT DEFAULT 0 COMMENT '是否禁用排名:0否1是',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_employee_no` (`employee_no`),
  KEY `idx_department` (`department_id`),
  KEY `idx_status` (`status`),
  KEY `idx_ranking_list` (`ranking_list_id`)
) ENGINE=InnoDB COMMENT='员工表';
```

#### 4.4.1 员工管理 `CurrentUserList.vue`

**路由**: `/user/current` | **文件**: `views/user/CurrentUserList.vue`

**数据模型**：与 DB `user` 表对齐，字段包含：
- 基础：id, employeeNo, name, gender, departmentId, position, rankLevel, hireDate, email
- 业务核心：**companyBelong**（归属组织 TD/XD/TY/WB 4种）、**rankingListId**（所属榜单）
- 扩展：education, baseLocation, projectRole, isRankingDisabled
- **status**：1在职 / 2禁用(不参与排名) / 3离职 / 4待岗 / 5停薪留职（5种状态）

> **v2.1 变更**：原"冻结"状态更名为"禁用"，语义为"暂时不参与积分排名"；对应的"启用"表示"加入积分排名，恢复生效"。

**筛选栏**：
- 搜索框（工号/姓名）
- 状态 下拉（在职/禁用/待岗/停薪留职，不含离职——离职走历史员工页）
- 归属组织 下拉（TD/XD/TY/WB）

**操作按钮**：
- 新增员工（Primary）
- 导出 Excel
- 模板下载
- 批量导入（a-upload）
- **字段管理**（齿轮图标按钮，打开字段配置抽屉）

**表格列**：工号 | 姓名 | 归属 | 部门 | 岗位 | 职级 | 榜单 | 积分 | 排名 | 状态(Tag) | 操作

> **v2.1 变更**：表格列支持通过"字段管理"功能进行列的显示/隐藏/排序，管理员可灵活调整列表展示字段。

**操作列**：
- 在职员工：编辑 | 禁用（需确认，提示"禁用后该员工暂不参与积分排名"） | 离职（需确认）
- 禁用员工：编辑 | 启用（恢复参与排名）
- 待岗/停薪留职：编辑 | 启用（恢复为在职）| 离职（需确认）

**新增/编辑弹窗**（720px, 双列布局）：
- 工号*, 姓名*, 归属组织*, 部门*(树选择), 性别, 入职日期, 邮箱, 学历(下拉), 岗位, 职级(P3-P12/M1-M6), base地, 所属榜单(下拉), 项目角色
- **员工状态**（下拉：在职/禁用/待岗/停薪留职，编辑时可见，新增时默认在职）

> **v2.1 变更**：编辑弹窗中增加状态字段，可直接修改为待岗、停薪留职等状态；表单字段列表未来支持通过"字段管理"进行扩展。

**字段管理抽屉**（Drawer, 400px）：
- 表格列字段列表，每行：字段名称 | 显示/隐藏(Switch) | 排序(拖拽或上下箭头)
- 保存按钮
- 预留：未来支持自定义字段(新增文本/下拉/日期等类型)

**分页**：20 条/页，底部分页器

#### 4.4.3 部门管理 `DepartmentManage.vue`

**路由**: `/department` | **文件**: `views/department/DepartmentManage.vue`

**布局**：左(7/24) 组织架构树 + 右(17/24) 部门详情 & 部门员工列表

**左侧 — 组织架构树**：
- a-tree 树形展示，default-expand-all
- 使用 `showLine` 连线模式 + `blockNode` 块节点
- 自定义 `#title` 插槽：节点卡片风格渲染，显示部门名称 + 层级Tag + 人数Badge
- 选中态高亮（蓝色左边框 + 浅蓝背景）
- 顶部"新增部门"按钮（Primary，齿轮图标）
- 点击节点选中，右侧联动刷新

> **v2.1 变更**：部门树使用 antd Tree 的 showLine + blockNode + 自定义 title 插槽，配合卡片式节点和连线，替代原始手写 div 列表，视觉更清晰美观。

#### 4.6.2 积分明细可见性规则

> 原始需求："录入人员或审核人员能看到该部分积分明细，不能看到其他积分明细，其他人只能看排名不能看明细"

| 角色 | 可见范围 |
|------|---------|
| 超级管理员 | 所有人的全部积分明细 |

| 普通员工 — 查看自己 | 自己的全部积分明细 |
| 普通员工 — 作为录入人 | 自己录入的那些积分记录（不论被评价人是谁） |
| 普通员工 — 作为审核人 | 被指定审核的那些积分记录 |
| 普通员工 — 查看他人 | **仅排名展示页的排名数据**，不可查看他人积分明细 |

**后端实现**：积分记录查询接口增加 `visibility` 过滤逻辑：
```
if role === 'employee':
    WHERE (user_id = currentUserId)         // 自己的记录
       OR (registrant_id = currentUserId)   // 自己录入的
       OR (auditor_id = currentUserId)      // 自己审核的
if role === 'dept_admin':
    WHERE user.department_id = currentUser.department_id
if role === 'super_admin':
    // 无限制
```

#### 4.6.3 已审核积分修改规则

> 原始需求："管理员能修改已审核通过的积分"

- **超级管理员**：可修改已审核通过的积分记录（编辑分值、指标、事由等）
- 修改时系统自动记录操作日志：`原值 → 新值 + 修改人 + 修改时间 + 修改原因（必填）`
- 修改后积分记录标记 `is_modified = true`，展示时显示「已修改」标签
- 修改后自动触发相关排名重新计算
- **部门管理员 / 普通员工**：不可修改已审核通过的积分

#### 4.6.4 部门数据隔离

> 原始需求："部门管理员不能看其他部门数据"

| 接口/页面 | 隔离方式 |
|----------|---------|
| 员工列表 | dept_admin 仅返回 `department_id = 本部门及子部门` 的员工 |
| 积分记录 | dept_admin 仅返回 `user.department_id` 匹配的记录 |
| 排名查看 | 全公司公开（所有角色均可查看排名列表），但明细受 4.6.2 约束 |
| 驾驶舱 | dept_admin 所有图表数据自动按 `departmentId` 过滤 |
| 报表导出 | dept_admin 仅可导出本部门相关报表 |

**后端实现**：NestJS 全局守卫 `DepartmentScopeGuard`，在 dept_admin 请求时自动注入 `departmentScope` 到查询条件，无需每个 Service 单独处理。

## 五、核心业务逻辑

### 5.1 过程分更新逻辑

```
每月维护流程:
1. 专人上传Excel或手动录入过程分
2. 系统对比: 被评价人 + 指标项 + 归属月份
3. 如存在上月记录 → 更新分值,历史存入process_point_history
4. 如不存在 → 新建记录
5. 当月排名计算时,只取当月最新过程分

计算公式:
人员当前总分 = 所有结果分总和 + 当月过程分最新值
```

### 5.2 排名计算逻辑

```
按需触发（"重新计算排名"按钮）或每日定时任务:
1. 按榜单分组计算
2. 对每个榜单内人员:
   - 计算结果分总和
   - 查询当月过程分
   - 总分 = 结果分 + 过程分
3. Redis缓存排名结果
4. 更新user表ranking字段
```

### 5.3 季度之星发布流程

```
1. 管理员选择季度
2. 点击"选定人员"逐一添加（指定榜单、排序、点评）
3. 可编辑展示顺序和点评内容
4. 点击"发布"将该季度之星公开
5. 发布后页面显示绿色成功提示
6. 可"撤回发布"回到编辑状态
```

## 十、注意事项

1. **数据安全**: 员工积分数据敏感，需做好权限控制（参见 4.6 权限与数据隔离规则）
2. **并发处理**: 排名计算时锁定，避免并发问题
3. **备份策略**: 数据库每日自动备份
4. **日志记录**: 关键操作（积分修改、审核、已审核积分变更）需记录操作日志
5. **性能优化**: 排名数据Redis缓存，数据库加索引
6. **驾驶舱性能**: 统计类查询使用物化视图或定时预计算，避免实时大量聚合查询
7. **AI 接口安全**: Phase 5 接入 LLM 时，需对传入数据脱敏，避免泄露员工隐私
