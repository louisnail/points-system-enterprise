> 创建日期：2026-04-14  
> 分支：`feature/team-health`  
> 状态：设计阶段，待开发

---

## 一、现状问题

### 1. 当前驾驶舱数据问题
- 所有数据为 **mock 硬编码**（`cockpit-data.ts`）
- 团队健康度评分固定为 76 分，维度得分写死
- 筛选条件（月份、部门、归属组织）**无实际数据联动**
- 刷新按钮仅显示提示，不请求真实数据

### 2. UI 失衡问题
- 图表高度固定 360px，无响应式适配
- SVG viewBox 固定 520x340，大屏/小屏显示不协调
- KPI 卡片固定 4 列布局，窄屏挤压

---

## 二、设计目标

### 核心原则
1. **统一标准**：各团队使用同一套健康度维度，便于横向对比
2. **灵活配置**：管理员可自定义维度、权重、评分规则
3. **数据驱动**：基于现有积分明细表计算，无需新增数据源
4. **实时响应**：筛选条件变化时，图表联动更新

---

## 三、系统架构

### 1. 双层架构设计

```
┌─────────────────────────────────────────────────────────┐
│                    团队健康度模板（全局唯一）              │
│  ┌─────────────────────────────────────────────────┐   │
│  │  维度1: 项目交付质量 (引用个人画像-管控项聚合)       │   │
│  │  维度2: 人均积分水平 (团队指标-人均计算)            │   │
│  │  维度3: 加分活跃度   (团队指标-趋势计算)            │   │
│  │  维度4: 人员稳定性   (团队指标-离职率计算)          │   │
│  └─────────────────────────────────────────────────┘   │
│                    ↓ 统一应用于所有团队                   │
└─────────────────────────────────────────────────────────┘
```

### 2. 维度类型定义

| 维度类型 | 说明 | 示例 |
|---------|------|------|
| **引用型** | 引用个人画像维度定义 | 项目交付质量、技术创新能力 |
| **团队指标型** | 基于团队整体数据计算 | 人均积分、参与率、离职率 |
| **对比型** | 与目标值/历史值对比 | 环比增长率、目标达成率 |

## 四、数据源分析

### 现有数据表支持

**积分明细表 `point_record`** 已有字段：
- `userId` - 被评价人
- `indicatorId` - 指标ID
- `score` - 确认分值（含正负）
- `pointStatus` - 积分状态（1过程分/2结果分）
- `auditStatus` - 审核状态（0待审核/1已通过/2已驳回）
- `belongMonth` - 归属月份（2026-03）

### 可计算指标

| 指标 | 计算逻辑 | 可行性 |
|------|---------|--------|
| 加分总数 | `SUM(score) WHERE score > 0 AND auditStatus = 1` | ✅ |
| 扣分总数 | `SUM(ABS(score)) WHERE score < 0 AND auditStatus = 1` | ✅ |
| 净积分 | `SUM(score) WHERE auditStatus = 1` | ✅ |
| 参与率 | `有积分记录人数 / 部门总人数` | ✅ |
| 录入条数 | `COUNT(*)` | ✅ |
| 人均积分 | `SUM(score) / COUNT(DISTINCT userId)` | ✅ |
| 待审核数 | `COUNT(*) WHERE auditStatus = 0` | ✅ |
| 环比增长率 | 本月 vs 上月对比 | ✅ |

**结论：所有指标均可从现有数据计算，无需新增数据表。**

## 五、数据库设计

```sql
-- 1. 团队健康度模板表（全局唯一，管理员配置）
CREATE TABLE team_health_template (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL COMMENT '模板名称',
  description TEXT COMMENT '说明',
  is_active TINYINT DEFAULT 1 COMMENT '是否启用',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 2. 团队健康度维度表
CREATE TABLE team_health_dimension (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  template_id BIGINT NOT NULL COMMENT '所属模板',
  name VARCHAR(100) NOT NULL COMMENT '维度名称',
  sort_order INT DEFAULT 0 COMMENT '排序',
  
  -- 数据源配置（JSON）
  data_source JSON NOT NULL COMMENT '数据源定义',
  
  -- 评分配置（JSON）
  scoring_config JSON NOT NULL COMMENT '评分方式配置',
  
  weight DECIMAL(5,2) DEFAULT 1.00 COMMENT '权重',
  
  -- 阈值配置
  threshold_excellent INT DEFAULT 90,
  threshold_good INT DEFAULT 70,
  threshold_warning INT DEFAULT 50,
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  FOREIGN KEY (template_id) REFERENCES team_health_template(id) ON DELETE CASCADE
);

-- 3. 团队健康度计算缓存表（按月存储，支持历史对比）
CREATE TABLE team_health_result (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  period VARCHAR(7) NOT NULL COMMENT '月份 2025-03',
  department_id BIGINT COMMENT '部门ID（null表示全公司）',
  ranking_list_id BIGINT COMMENT '榜单ID（null表示全部）',
  org_code VARCHAR(20) COMMENT '归属组织（null表示全部）',
  
  overall_score INT COMMENT '综合得分',
  
  -- 各维度得分（JSON）
  dimension_scores JSON COMMENT '{dimId: score, ...}',
  
  -- 原始数据快照（用于追溯）
  raw_data JSON COMMENT '计算用的原始数据',
  
  calculated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  UNIQUE KEY uk_period_scope (period, department_id, ranking_list_id, org_code)
);
```

## 七、筛选联动设计

### 筛选条件
- **时间维度**：月度/季度/年度
- **具体月份**：2026-03
- **部门筛选**：技术研发部/交付管理部/...
- **归属组织**：TD/XD/TY/WB

### 联动逻辑
```typescript
// 筛选条件变化时，重新计算所有维度
watch([timeDimension, selectedMonth, filterDept, filterOrg], () => {
  // 1. 构建查询参数
  const query = buildQueryParams();
  
  // 2. 调用后端聚合API
  fetchHealthData(query).then(data => {
    // 3. 按配置的维度公式计算得分
    const scores = calculateDimensionScores(data, dimensionConfigs);
    // 4. 更新图表
    updateCharts(scores);
  });
});
```

### 性能优化
| 策略 | 说明 |
|------|------|
| 预计算 | 每月1号凌晨预计算上月各维度数据 |
| 缓存 | 相同筛选条件缓存10分钟 |
| 分页聚合 | 部门/榜单维度先聚合再计算 |
| 异步加载 | 各图表独立加载，避免阻塞 |

## 九、开发方案对比

| 方案 | 工作量 | 特点 |
|------|--------|------|
| **完整版**（灵活配置） | 6天 | 支持自定义维度、权重、评分规则 |
| **简化版**（固定维度） | 3天 | 5个维度固定，权重写死，只做展示 |
| **最简版**（仅展示） | 1.5天 | 配置直接改数据库，前端只做展示 |

### 推荐：简化版（3天）

固定5个维度：
1. 项目交付质量（引用个人画像）
2. 人均积分水平（团队指标）
3. 加分活跃度（趋势计算）
4. 参与覆盖率（人员指标）
5. 人员稳定性（离职率）

**好处**：
- 工作量减半
- 够用了（各团队用同一套标准）
- 后期需要再扩展
